《Project: Memory.log》每轮开发后的最小测试要求

文档定位：
本文件定义每一轮 AI 开发结束后，最少必须执行哪些测试、如何根据测试结果决定下一步，以及在无法测试时如何记录。

本文件的目标是提升自动化开发可靠度。
规则核心不是“尽可能多测”，而是“每次改完都做与本轮改动匹配的最小必要验证”。
并且测试要求默认同时面向两类对象：
1. AI 自动验证
2. 人类上手验证

--------------------------------------------------
一、总原则
--------------------------------------------------

每次 AI 开发结束后，不能只说“代码已改完”，必须回答以下四个问题：

1. 我这轮改了什么
2. 这些改动最容易把什么弄坏
3. 我实际跑了哪些测试
4. 根据测试结果，下一步应该是继续推进、先修 bug，还是先补测试设施
5. 这次改动有没有做成“人能看懂是否生效”的状态

--------------------------------------------------
二、所有开发轮次都必须做的最低检查
--------------------------------------------------

不管本轮改动是什么，至少都要做以下检查：

1. 项目 / 场景加载检查
目标：
确认改动没有造成明显启动错误、资源丢失或脚本报错。

2. 改动文件相关的静态检查
目标：
确认节点路径、资源引用、导出参数和脚本逻辑没有明显断裂。

3. 交接记录
目标：
明确写出：
- 实际跑了什么
- 没跑什么
- 为什么没跑

4. 人类可见反馈检查
目标：
确认本轮新增或修改的关键机制，不仅可被脚本读取，也有最小白盒可见反馈，能让人快速判断“是否触发、何时结束、影响范围是什么”。

如果这些最低检查没有做完整，本轮开发不应视为可靠完成。

--------------------------------------------------
三、按改动类型匹配的最小测试要求
--------------------------------------------------

A. 改了玩家动作 / 层切换逻辑
至少要测：
1. 层切换基础验证
2. 玩家关键状态是否正常变化
3. 对应动作的最小场景验证
4. 玩家是否有可见反馈让人能看懂动作已触发

若改动涉及：
- `player_g.gd`
- `player_g.tscn`
- 动画状态机
- 层碰撞掩码

则默认必须跑：
- TC-01 玩家层切换基础验证

--------------------------------------------------

B. 改了投射物 / 风道 / A -> B 切层逻辑
至少要测：
1. 投射物是否能正常生成
2. 切层后是否正确转化
3. Buff / 状态变化是否正确生效

若改动涉及：
- `bullet.gd`
- `bullet.tscn`
- A -> B 转化逻辑

则默认必须跑：
- TC-02 子弹切层转风道验证

--------------------------------------------------

 C. 改了大锤 / B -> A 带怪逻辑
至少要测：
1. 命中是否触发
2. 敌人是否正确被带回
3. 状态是否保留
4. 玩家是否能继续战斗
5. 人类是否能通过白盒提示或视觉变化区分“留层 / 切层”

若改动涉及：
- `hammer_hitbox.gd`
- B 层组合切技带怪
- B 层重击切层

则默认必须跑：
- TC-03 B 层组合切技带回单体敌人

如果本轮改的是群体带回，还应补测：
- TC-05 B 层重击切层群体带回

--------------------------------------------------

D. 改了竞技场 / 波次 / 场景触发器
至少要测：
1. 触发是否成功
2. 门是否正确开关
3. 波次是否正确结算

若改动涉及：
- `arena_trigger.gd`
- 竞技场 section
- 刷怪和开关门逻辑

则默认必须跑：
- TC-04 竞技场开关门流程验证

--------------------------------------------------

E. 改了 A 层踢怪 / 强制挂起 / 权限解锁
至少要测：
1. 目标是否能被送去 B 层
2. 层级归属是否正确变化
3. 目标切层后状态是否保留
4. 权限解锁后相关能力是否真的开放

若改动涉及：
- A 层踢怪逻辑
- `can_switch_layer`
- `can_use_hammer`
- 权限解锁触发器或解锁段

则默认必须跑：
- TC-07 A 层踢怪 / 强制挂起验证

若改动已接入章节路线，还应补测：
- TC-08 第二章前 10 分钟章节冒烟测试

--------------------------------------------------

F. 改了敌人 AI / 敌人层级行为
至少要测：
1. 敌人能否正常生成
2. 基础行为是否运行
3. 在目标层级的行为差异是否仍成立
4. 是否影响既有切层逻辑
5. 关键状态变化是否有最小可见反馈

如果测试场景尚未完善，至少要做：
1. 单独生成该敌人的场景加载验证
2. 与玩家的最小交互验证

如果涉及主观体验疑点，还应登记：
- `docs/AI持续开发/12_人工试玩待测清单.txt`

--------------------------------------------------

G. 改了 section / 关卡结构 / TileMap / 摆放
至少要测：
1. 场景能否打开
2. 关键 section 是否能进入
3. 玩家不会开局卡进碰撞体或掉出地图
4. 相关触发器仍能触发

若改动范围较大，还应做：
1. 章节冒烟测试

--------------------------------------------------

H. 改了测试基础设施本身
至少要测：
1. 测试场景能否加载
2. PASS / FAIL / BLOCKED 日志是否正确输出
3. 测试脚手架没有把正式逻辑污染坏

--------------------------------------------------
四、根据测试结果怎么处理
--------------------------------------------------

测试结果不能只记录“过了”或“没过”，而必须导向下一步动作。

情况 1：全部通过
下一步：
可以进入下一项开发任务。

记录要求：
在交接日志写明：
- 通过了哪些测试
- 本轮改动可继续叠加开发

--------------------------------------------------

情况 2：有失败项，但问题明确、范围可控
下一步：
优先修复失败项，不应直接叠加更多功能。

记录要求：
在交接日志写明：
- 哪项测试失败
- 失败原因推测
- 下一轮应优先修哪里

任务看板要求：
把对应修复项写入 `进行中` 或 `待做` 顶部。

--------------------------------------------------

情况 3：测试阻塞，因为功能未实现完或测试设施缺失或 Godot 无法执行
下一步：
不能假装通过。
必须明确判断是：
1. 功能还没完，先补功能
2. 测试设施缺失，先补测试基础设施
3. 环境无法执行，先恢复可运行环境

记录要求：
在交接日志写明：
- 哪项测试因何阻塞
- 需要补哪个测试场景、脚手架或环境前提

遗留问题要求：
如果这会影响连续开发，需同步写入 `06_遗留问题与风险.txt`

--------------------------------------------------

情况 4：启动级检查都失败
下一步：
本轮开发不应继续叠加新内容，必须先恢复可运行状态。

记录要求：
在交接日志明确标记：
`当前状态不可继续叠加开发`

--------------------------------------------------

情况 5：自动测试已做完，但仍有主观体验疑点
下一步：
把相关项写入 `12_人工试玩待测清单.txt`，并可继续推进不依赖该结论的结构化任务。

记录要求：
在交接日志中增加 `MANUAL` 项，写明待人确认的问题。

补充说明：
“仍有主观体验疑点”并不代表可以只留内部读数。
只要本轮改动与玩家体验有关，就应先补最小可见反馈，让人工后续能更快判断问题。

--------------------------------------------------
五、无法测试时的最低记录格式
--------------------------------------------------

如果因为条件不足无法测试，交接日志至少必须写：

1. 本应跑的测试：
2. 未跑原因：
3. 当前是否存在高风险盲区：
4. 下轮要先补什么才能测：

禁止写法：
- “未测试”
- “理论上可行”
- “应该没问题”

这些都不足以支持自动化连续开发。

--------------------------------------------------
六、每轮开发结束的标准记录模板
--------------------------------------------------

建议每轮开发结束后，在交接日志追加以下结构：

[本轮测试]
必须测试：
1.
2.

实际执行：
1.
2.

结果：
1. PASS:
2. FAIL:
3. BLOCKED:
4. MANUAL:

测试结论：
1. 可继续下一任务
或
2. 应先修复失败项
或
3. 应先补测试基础设施
或
4. 应先恢复可运行环境

--------------------------------------------------
七、当前阶段的首批强制测试要求
--------------------------------------------------

在第二章当前阶段，只要相关功能存在，以下测试应尽快成为强制项：

1. TC-01 玩家层切换基础验证
2. TC-02 子弹切层转风道验证
3. TC-03 B 层组合切技带回单体敌人
4. TC-04 竞技场开关门流程验证
5. TC-07 A 层踢怪 / 强制挂起验证

它们是第二章自动化可靠性的最小闭环。
