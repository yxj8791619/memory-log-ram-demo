《Project: Memory.log》项目管理进度汇报

最后更新时间：
2026-03-28（补充小节点完成后本地 git 提交规则）

文档用途：
这份文档不是给开发者做技术交接用的，而是给项目管理人员快速看进度、看风险、看下一步。
因此写法必须坚持一个原则：
“给人看的”。

这句话的含义是：
1. 先说结论，再说细节
2. 先说进度、影响、风险、下一步，再说技术实现
3. 尽量少用只有开发者才懂的缩写、内部术语和代码细节
4. 任何 AI 在达到一个小节点后，都要顺手维护这份文档，不要攒很多轮再一起补

这里的“小节点”包括但不限于：
1. 一条新规则补齐
2. 一个 bug 修掉
3. 一个骨架搭好
4. 一轮测试完成
5. 一个阻塞被识别
6. 一个阻塞被解除
7. 一个阶段目标被确认推进或推迟

--------------------------------------------------
四、当前最主要风险
--------------------------------------------------

1. 虽然权限解锁导流已补齐，但风道教学和综合战的最终读场效果仍需要人工试玩确认
2. 虽然双向单体切层和第一次综合战已可验证，但群体切层和独立功能怪还没进入可验证状态
3. 新规则虽然已经拆开，但教学提示和群体切层还没跟上，仍有误学成“所有 B 层招都能切回”的风险
4. 如果后续执行时不严格按“小节点一提交”落地，文档闭环虽然更清楚，但开发过程仍可能再次出现多个改动混在同一提交里的追溯问题

--------------------------------------------------
五、对项目管理最重要的结论
--------------------------------------------------

1. 项目已经从“只有结构骨架”推进到“前 10 分钟主循环、双样板怪、第一次综合战和权限解锁导流都可测试”
2. 当前最合理的管理判断，是继续收束“普通锤击留层 / 锤攻击 + 切层键带回”的教学表达；关卡几何、切层落地和锤击命中现在都已经有基础回归兜底
3. 只要继续保持“新机制或新关卡段落落地就补测试”的节奏，后续 AI 自动开发的可控性仍在上升
4. 现在连开发过程本身也补上了最小提交纪律，后续定位“某个小节点具体改了什么”会更容易
