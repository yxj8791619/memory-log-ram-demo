《Project: Memory.log》AI 开发协议与回写规范

--------------------------------------------------
一、工作模式
--------------------------------------------------

AI 每次接手开发时，必须按以下顺序执行：

1. 阅读本目录入口文档
2. 阅读当前状态、任务看板、遗留问题、最近交接日志
3. 检查代码与场景当前真实状态
4. 只选择一个主目标推进
5. 完成后更新本目录文档
6. 达到任何小节点后，同步维护项目管理进度汇报文档
7. 每完成一个小节点后，立即做一次本地 git 提交

补充说明：
这里的“小节点”包括规则补齐、小修复、骨架建立、一轮测试完成、阻塞出现或解除等。

维护 `docs/AI持续开发/15_项目管理进度汇报.txt` 时，原则必须是“给人看的”：
1. 优先写结果、影响、风险和下一步
2. 不要写成只对开发者有意义的技术流水账
3. 只保留阶段级信息，不追着每个小提交累加

每个小节点完成后的提交要求如下：
1. 提交只覆盖当前这个小节点，不要把多个不同目的的修改混在一起
2. commit message 必须简短清楚，并带任务类型关键字
3. 任务类型关键字至少使用以下之一：`fix`、`feat`、`docs`、`test`、`refactor`
4. 推荐格式：`fix: 修正切层掉落`、`feat: 新增功能怪样板`、`docs: 补充提交规则`

并且要始终遵守“双重可测”原则：
1. 便于 AI 自动测试
2. 便于人类上手测试

任何机制、状态、触发器、切层派生、Buff、Debuff、护盾、范围判定，都不要只做“脚本里可读”。
至少还要补一种人类能快速看懂的白盒反馈，例如：
1. 屏幕提示
2. 颜色 / 透明度变化
3. 节点显隐
4. 白盒特效或标记
5. 调试文本

不要一上来同时推进多个大系统。

--------------------------------------------------
二、主目标选择规则
--------------------------------------------------

每一轮开发，优先做以下类型任务：

1. 能直接提升“可玩性”的任务
2. 能让某条设计结论被真机验证的任务
3. 能为后续自动开发减少歧义的任务

当前统一优先级顺序固定为：
1. 项目结构 AI 友好化改造
2. AI 可执行测试体系改造
3. test_room / chapter2 白盒内容开发

每轮开发尽量只选一个主目标，外加少量配套修复。

额外流程闸门：
1. 如果连续两轮都只在同一场景里做“提示文案 / 标签位置 / 白盒信号位置”级别的小修补，第三轮禁止继续同类微调
2. 出现上述情况时，下一轮必须改做以下三类动作之一：整理人工试玩关注点、登记系统性风险、或做一次成组的结构性收束
3. 同一轮里，如果改动目标仍是同一教学段，同类微调小节点最多记 2 个，不要把 5 个近似动作拆成 5 次连续提交
4. 如果同一批测试连续通过、但人工待测疑点持续增加，优先级应切到“人工试玩准备 / 问题归并”，而不是继续堆白盒信号
5. 如果当前目标已经进入“需要人工判断是否更自然”的阶段，AI 不应继续仅凭文案和标签做细碎优化

--------------------------------------------------
三、禁止事项
--------------------------------------------------

1. 不要擅自推翻第二章核心规则
2. 不要在未验证主循环前过度扩张怪物池
3. 不要在没有记录的情况下删除重要文档
4. 不要只改代码不更新交接文档
5. 不要把“设计讨论”当成“实现完成”
6. 不要跳过与本轮改动类型匹配的最小必要测试
7. 不要把“人工待测”误写成“测试通过”
8. 不要在完成小节点后跳过本地 git 提交
9. 不要只补内部状态，不补人类可见反馈
10. 不要把同一教学段的连续微调写成越来越长的流水账
11. 不要把项目管理汇报写成按提交顺序罗列的开发日记
12. 不要让自动测试过度绑定单句文案或像素级位置，除非该文案或位置本身就是本轮唯一目标

--------------------------------------------------
四、允许 AI 自主决策的范围
--------------------------------------------------

AI 可以自主决定：

1. 代码重构方式
2. 白盒临时实现形式
3. 小型数值调整
4. 目录内持续开发文档的补充与整理
5. 在不改变核心设计的前提下，对实现细节进行简化

AI 不应自主决定：

1. 推翻第二章双层主循环
2. 修改 A/B 两层职责分工
3. 把“都能切层”改回“部分能切层”
4. 删除整条动作线或整类怪物设计，除非已记录原因

--------------------------------------------------
五、每轮开发结束必须回写的文件
--------------------------------------------------

至少更新以下文件：

1. docs/AI持续开发/03_第二章当前状态.txt
2. docs/AI持续开发/05_任务看板.txt
3. docs/AI持续开发/07_交接日志.txt
4. docs/AI持续开发/15_项目管理进度汇报.txt

如果发现新的系统性风险，还必须更新：

5. docs/AI持续开发/06_遗留问题与风险.txt

如果出现新的人工体验问题、节奏疑点或手感疑点，还必须更新：

6. docs/AI持续开发/12_人工试玩待测清单.txt

并且必须参考：

7. docs/AI持续开发/11_每轮开发后的最小测试要求.txt

--------------------------------------------------
六、回写格式要求
--------------------------------------------------

每次开发结束，在交接日志中追加一条记录，格式固定为：

[日期]
本次主目标：

完成内容：
1.
2.

修改文件：
1.
2.

验证情况：
1.
2.

未完成 / 遗留：
1.
2.

下一步建议：
1.
2.

[本轮测试]
必须测试：
1.
2.

实际执行：
1.
2.

结果：
1. PASS:
2. FAIL:
3. BLOCKED:
4. MANUAL:

测试结论：
1. 可继续下一任务 / 应先修失败项 / 应先补测试基础设施

说明：
`MANUAL` 用于记录必须由人后续体验确认、但不阻塞 AI 继续推进的事项。

项目管理汇报附加要求：
1. 每次达到任何小节点后，也要同步维护 `docs/AI持续开发/15_项目管理进度汇报.txt`
2. 写法优先让项目管理人员快速读懂“现在做到哪、影响是什么、风险是什么、下一步做什么”
3. 不要把项目管理汇报写成与交接日志重复的技术细账
4. `15_项目管理进度汇报.txt` 默认只保留阶段级信息
5. 管理汇报中的“已完成”建议控制在 8 条以内，“主要风险”建议控制在 5 条以内，“下一步”建议控制在 4 条以内

git 提交附加要求：
1. 每次达到任何小节点后，也要立刻提交一次本地 git
2. commit message 保持简短，不写成长段说明
3. commit message 必须用任务类型关键字开头，例如 `fix:`、`feat:`、`docs:`、`test:`、`refactor:`
4. 如果多个提交都属于同一场景的同类微调，最多允许连续 2 个；再往后必须先停下来归并问题
5. 如果 Godot 为正式受管脚本生成了新的 `.gd.uid` 文件，默认应一并纳入版本控制；不要长期让它们游离在工作区里
6. 只有在对应脚本本身属于临时实验文件、或明确不会长期保留时，才不要提交对应 `.uid`

--------------------------------------------------
七、当前状态文件更新要求
--------------------------------------------------

03_第二章当前状态.txt 必须始终保持简洁、准确、可快速阅读。
更新时优先改这几项：

1. 当前开发阶段
2. 当前已完成
3. 当前进行中
4. 下一优先任务
5. 当前阻塞项

--------------------------------------------------
八、任务看板更新要求
--------------------------------------------------

05_任务看板.txt 中的任务状态只允许使用以下四种：

1. `待做`
2. `进行中`
3. `已完成`
4. `阻塞`

更新时不要丢历史上下文，但也不要让看板膨胀到不可读。

额外要求：
1. 看板顶部的唯一推荐动作，必须优先指向“最能降低不确定性”的动作，而不是“最容易继续微调”的动作
2. 如果某项内容已经明显进入人工判断区，不要继续把它长期挂在“进行中”却只做局部小修补

--------------------------------------------------
九、遗留问题文件更新要求
--------------------------------------------------

06_遗留问题与风险.txt 只记录以下内容：

1. 尚未定论的问题
2. 当前临时实现的隐患
3. 需要未来某轮单独验证的假设
4. 会影响自动化开发连续性的注意事项

不要把普通任务堆在这里。

如果发现以下情况，也应记为流程风险：
1. 同一教学段在短时间内被连续细碎修改
2. 管理汇报和交接日志开始明显膨胀
3. 测试开始过度耦合具体文案或局部摆位

--------------------------------------------------
十、人工试玩待测清单更新要求
--------------------------------------------------

12_人工试玩待测清单.txt 只记录以下内容：

1. AI 不能独立判定的问题
2. 需要人实际游玩后给结论的问题
3. 不应阻塞自动开发、但应等待人工反馈的问题

不要把已可自动验证的功能回归项写进这里。

补充：
“需要人工试玩确认”不等于“可以不做可见反馈”。
即使某项结论仍需人工体验判断，AI 也应先把它做成“人能快速看懂是否触发”的状态。
同一主题的人工试玩疑点应尽量合并记录，不要因为同一教学段里新增 3 个相近标签，就拆成 3 条高度重复的待测项。

--------------------------------------------------
十一、当你不知道下一步做什么时
--------------------------------------------------

按以下顺序判断：

1. 任务看板里是否有 `进行中`
2. 当前状态里的“下一优先任务”是什么
3. 当前阻塞项是否已经解除
4. 当前统一优先级是否仍未完成结构改造或测试改造

如果以上都不清楚，默认回到：
“先做项目结构 AI 友好化改造；若结构底盘已具备，则先补最小测试基础设施；再推进白盒内容。”

补充：
一旦进入代码实现阶段，还必须同步参考：
`docs/AI持续开发/11_每轮开发后的最小测试要求.txt`
